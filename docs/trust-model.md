# OCC Trust Model

**Canonical claim:**

> OCC guarantees single-successor semantics within the verifier-accepted measurement and monotonicity domain of the enforcing boundary.

## What This Means, Precisely

A *successor* is a proof that a verifier accepts as valid under a given `VerificationPolicy`.

*Single-successor semantics* means: within the measurement and counter domain defined by the verifier's policy, at most one accepted successor can exist per committed state.

This guarantee holds if and only if:

1. The enforcing boundary is correctly isolated (see [Assumptions](#assumptions))
2. The verifier policy is strict â€” `allowedMeasurements` is pinned and `minCounter` is enforced
3. The signing key has not been exfiltrated from the boundary

---

## Assumptions

| # | Assumption | Failure consequence |
|---|-----------|-------------------|
| 1 | **Boundary isolation** â€” the TEE or equivalent correctly prevents external processes from accessing the signing key or submitting alternative commit paths | Boundary compromise collapses all guarantees |
| 2 | **Key secrecy** â€” the Ed25519 private key is generated inside the boundary and never leaves it | Key exfiltration allows proof forgery |
| 3 | **Nonce freshness** â€” each nonce is generated by the boundary from a fresh, unpredictable source (â‰Ą128 bits); nonce reuse is a protocol violation | Nonce reuse allows replay within a session |
| 4 | **Honest measurement** â€” for `measured-tee` tier, the attestation report is produced by hardware that correctly measures the enclave binary | Delegated to TEE vendor (AWS Nitro, Intel SGX, etc.) |
| 5 | **Monotonic counter durability** â€” the counter is durable across process restarts for backends that advertise this property | Ephemeral counters degrade anti-rollback to within a single session |
| 6 | **Strict verifier policy** â€” the caller pins `allowedMeasurements` and enforces `minCounter`; a permissive policy degrades the guarantee | See [Threat Model](#threat-model) |

---

## Non-Goals

- **Global ordering** â€” OCC does not establish a total ordering across independent boundaries. Two separate boundaries can produce proofs with overlapping counters.

- **Cross-boundary double-spend prevention** â€” OCC does not prevent the same artifact from being submitted to two different enforcing boundaries. Cross-boundary coordination requires an external mechanism.

- **Byte-level copy prevention** â€” OCC does not prevent the raw bytes of an artifact from being copied. It prevents the *authoritative right* â€” the accepted proof lineage â€” from being duplicated within a policy domain.

- **Consensus replacement** â€” OCC is not a substitute for distributed consensus among mutually distrustful parties. It constrains what a single enforcing boundary can produce.

- **Notary honesty** â€” In remote notary configurations, OCC does not guarantee the notary is honest. It guarantees the proof is independently verifiable offline using only the public key and hash lineage.

- **Metadata integrity** â€” The `metadata` field is advisory and not signed. It must not be used for security decisions.

## Threat Model

### In-Scope Threats

| Threat | Mechanism | Mitigation |
|--------|-----------|-----------|
| Proof replay | Attacker presents an old but valid proof as current | `minCounter` in `VerificationPolicy` rejects proofs whose counter is below the last accepted value |
| Measurement substitution | Attacker claims a weaker enforcement boundary | `allowedMeasurements` pins exact PCR0/MRENCLAVE values; mismatches reject |
| Signature forgery | Attacker fabricates a proof without the signing key | Ed25519 unforgeability under ECDLP hardness |
| Downgrade attack | Attacker strips `measured-tee` and substitutes `stub` | Enforcement tier is signed; `requireEnforcement` in policy rejects weaker tiers |
| Post-hoc authorization wrapping | Attacker signs an artifact after the fact using a separate key | Proof is produced on the commit path â€” no commit means no proof |
| Chain gap insertion | Attacker deletes a link from a proof chain | `prevB64` chaining â€” any removed link breaks hash continuity; application layer detects |
| Metadata tampering | Attacker modifies `metadata` field | `metadata` is unsigned; verifier consumers must ignore it for trust decisions |

### Out-of-Scope Threats

| Threat | Why out of scope |
|--------|----------------|
| Signing key exfiltration from boundary | Requires successful boundary compromise â€” assumed secure |
| TEE firmware vulnerability | Delegated to hardware vendor |
| Verifier supplied with weak policy | Caller responsibility; the verifier enforces exactly what policy declares |
| Physical access to enclave host | Outside OCC's threat model |
| Two independent boundaries colluding | Requires cross-boundary coordination mechanism |

## "If Two Successors Appear" Diagnostic

If two distinct verifier-accepted proofs exist for the same counter value within the same measurement domain, at least one of the following must be true:

1. **Key exfiltration** â€” the signing key was extracted from the boundary and used to sign a second proof outside it.
   *Diagnostic indicator:* both proofs carry valid signatures for the same `publicKeyB64` and `measurement`, different `nonceB64`.
   *Mitigation:* hardware attestation + physical security of the enclave host.

2. **Counter implementation bug** â€” the counter returned the same value for two distinct commits.
   *Diagnostic indicator:* two proofs with identical `counter` values, different `nonceB64`.
   *Mitigation:* monotonic counter with durable persistence and write-before-return semantics. See [`packages/stub/src/stub-host.ts`](../packages/stub/src/stub-host.ts).

3. **Policy weakness** â€” the verifier accepted both proofs under a policy that does not enforce `minCounter` or `allowedMeasurements`.
   *Diagnostic indicator:* policy has no `minCounter`; old proofs still pass.
   *Mitigation:* strict policy (see [`src/types.ts` â€” `VerificationPolicy`](../src/types.ts)).

4. **Boundary bypass** â€” an alternative commit path exists outside the declared boundary.
   *Diagnostic indicator:* proofs with same `measurement` but produced by different code paths.
   *Mitigation:* audit boundary isolation â€” this is a structural implementation failure, not a cryptographic one.

5. **Measurement collision** â€” two distinct boundaries share the same `measurement` string.
   *Diagnostic indicator:* same measurement, different physical deployments.
   *Mitigation:* use vendor-attested measurements (PCR0, MRENCLAVE) that are deterministically derived from the enclave binary.

## Enforcement Tiers

| Tier | Enforcement | Counter durability | Use case |
|------|-------------|-------------------|----------|
| `stub` | Software only, key in process memory | Session (ephemeral) or persistent (file-backed) | Development, testing, local demos |
| `hw-key` | Hardware-protected signing key (HSM, Secure Enclave); commit gate runs outside measured boundary | Adapter-defined | Key custody without code measurement |
| `measured-tee` | Full causal enforcement inside attested enclave; host treated as untrusted | Hardware monotonic counter (adapter-defined) | Production, highest assurance |

The `enforcement` field is signed and tamper-evident in transit. However, it is **self-reported** by the adapter. A verifier requiring `requireEnforcement: "measured-tee"` in policy will reject proofs claiming a weaker tier â€” but `requireEnforcement` alone is not sufficient. It **must** be combined with `allowedMeasurements` (and `requireAttestation` for `measured-tee`) to establish actual trust.
