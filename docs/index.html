Origin Controlled Computing

🔐 Proof should be a property of creation, not verification.

Today, digital artifacts are created freely and verified later. OCC introduces a different model: valid proof exists only if specific bytes were committed through an authorized execution boundary during a single atomic event.

The Problem

⚠️ Computers create data first, then try to prove it later.

Photos, documents, sensor outputs, logs, model responses, and files are usually created without structural proof. Only afterward do we attach signatures, metadata, timestamps, registries, or credentials.

By then, the artifact already exists. Proof becomes optional. Trust becomes something we attempt to recover after creation instead of something the system enforced at the moment the artifact came into being.

The Shift

🔄 OCC moves math to the commit path.

Under OCC, proof is not wrapped around an artifact after the fact. It is produced only when candidate bytes cross an authorized execution boundary and are committed through a protected path.

If proof exists, the authorized commit path was traversed.
How It Works

⚛️ One atomic event.

OCC binds authorization, cryptographic binding, and commit into a single indivisible operation. Not a later check. Not a label. A creation event.

01

Authorize

Candidate bytes enter a protected commit interface controlled by an authorized execution boundary.

02

Bind

Inside the boundary, the system computes a content-dependent hash and combines it with boundary-fresh cryptographic output.

03

Commit

The artifact and its proof are committed together. If the event does not occur, no valid proof can exist.

Proof is caused, not added.

What Changes

🧱 Proof becomes structural.

Before OCC

Create first.

Artifacts are created freely. Verification is applied later. The trust model begins after the artifact already exists.

With OCC

Commit with proof.

Valid proof exists only because the system made it reachable through a specific authorized boundary and commit path.

Why It Matters

🌍 A new primitive for trustworthy digital systems.

OCC applies anywhere digital artifacts must be trusted as outputs of a particular creation path rather than merely inspected after creation. It is not proof of truth. It is proof that specific bytes were committed by a specific boundary under a specific authorization event.

AI Outputs
LLM responses, generated images, model predictions. Prove which model, version, and prompt produced a specific output.
Journalism & Media
Photos, video, audio recordings. Prove a file was captured by a specific device at a specific time, not fabricated later.
Scientific Instruments
Sensor readings, lab equipment data, telescope imagery. Prove raw data came from an authorized instrument, unmodified.
Compliance Logs
Audit trails, financial records, regulatory filings. Prove log entries were created by the system at the recorded time.
Software Pipelines
Build artifacts, deployment packages, signed releases. Prove a binary was produced by a specific CI/CD pipeline run.
Digital Identity
Credentials, attestations, signed claims. Prove an identity artifact was issued by an authorized authority through a controlled path.
